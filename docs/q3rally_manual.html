Overview

Game Modes

Q3Rally ships with twelve distinct game modes covering pure racing, armed racing, team play, survival, and arena combat. The g_gametype server variable selects the active mode.

✔ v0.7c FixAll gametype values now have explicit integer assignments in bg_public.h. Non-team modes retain historical values (0–6). Team modes start at 16, leaving slots 8–15 for future additions.

Server Setup

Q3Rally dedicated servers are configured via .cfg files. The most important variable is g_gametype which selects the active game mode.

Starting the Server

PlatformCommand
Windows (manual)q3rally.x86_64.exe +set dedicated 1 +exec racing_server.cfg
Windows (.bat)Double-click baseq3r/start_racing_server.bat — picks the correct executable automatically. Each multiplayer gametype has its own batch file.
Linux./ioq3rallyded.x86_64 +set dedicated 1 +exec racing_server.cfg
NoteGT_SINGLE_PLAYER is local-only. Its batch file notes that a dedicated server is not available for this mode.

Online Ranking

Ladder Service

The Q3Rally Ladder automatically records match results from dedicated servers and publishes leaderboards per game mode and map. The ladder web frontend is available at ladder.q3rally.com.

How It Works

1
Match ends on the server
When intermission starts, sv_ladder.c assembles a JSON payload from the level state — timestamps, player positions, lap times, kills, and damage figures.
2
POST to the ladder endpoint
The server posts to POST /api/v1/matches using libcurl asynchronously. The request includes a bearer token from sv_ladderApiKey.
3
Retry on failure
Transport errors or HTTP ≥ 500 queue the payload to telemetry/outbox and retry with exponential backoff (5 s → 5 min cap). Successful delivery removes the spool entry.
4
Leaderboard updated
The ladder service indexes the match and updates best times, K/D records, and objective scores. The web frontend reflects changes within seconds.

Server CVars

CVarDefaultFlagsDescription
sv_ladderEnabled0ARCHIVEGlobally enable or disable the HTTP reporter. Set to 1 to activate.
sv_ladderUrl—ARCHIVETarget endpoint. Example: https://ladder.q3rally.com/index.php
sv_ladderApiKey—TEMP · PROTECTEDBearer token for authentication. Never appears in status output or logs.
Security Notesv_ladderApiKey is treated like rconPassword — stored as a temporary protected CVar that is never included in status responses or written to disk.

Mode-Specific Payload Notes

Race Modes (GT_RACING, GT_RACING_DM, GT_SPRINT)

Include raceStart, raceFinish, completedLaps, bestLap, and totalTime. The DM variant adds damageDealt/damageTaken per player.

Team Racing (GT_TEAM_RACING, GT_TEAM_RACING_DM)

Team lap times are stored in level.teamTimes and normalized by participant count. Player fields mirror solo racing, extended with team membership.

Elimination & LCS

Adds eliminationSchedule (start delay, interval, warning), survivalTime, elimOrder, and lapsAtElim per player.

Derby

Uses race timing fields combined with knockout and damage efficiency metrics via score_t.score, damageDealt, and damageTaken.

CTF / CTF4

Flag metrics: captures, assistCount, defendCount, and carrier-related damage values. Team scores via level.teamScores.

Domination / KOTH

Control-point ticks from level.teamScores/level.teamTimes, paired with per-player score_t.score and damage figures.

Transport & Reliability

The HTTP façade lives in engine/code/server/sv_ladder.c and uses a dedicated CURLM handle for fully asynchronous POST requests. Communication uses libcurl (USE_CURL / USE_CURL_DLOPEN).

BehaviourDetail
Retry backoffTransport errors or HTTP ≥ 500 → queued to telemetry/outbox. Retry: 5 s → exponential → max 5 min.
Spool limitControlled by sv_telemetryMaxBatch. Stored under fs_homepath.
TLSUses system trust store by default. Certificate failures trigger backoff without marking the match as delivered.
Authsv_ladderApiKey sent as Authorization: Bearer .... Never logged.
DeduplicationServer-side deduplication on matchId — re-posting the same ID is a no-op.

Atmospheric Weather — Mapper Guide

Practical checklist for placing and tuning weather volumes in Q3Rally maps.

#TopicNotes
1 Place weather entities correctly Use rally_weather_rain for rain volumes and rally_weather_snow for snow. Both are created as ET_WEATHER server-side. Brush bounds matter — the client checks whether particle positions are inside the volume.
2 Type wiring rally_weather_rain sets weapon = 0, rally_weather_snow sets weapon = 1. The client uses this value to select the weather type. Baseline particle count is carried in entity state.
3 Rain spawnflags For rain, spawnflags are passed to the client as a splash toggle: controls whether waterSplash / landSplash effects are enabled. Snow splash is disabled by design regardless of spawnflags.
4 Adaptive quality Particle density scales automatically per client. Profile is selected via cg_atmosphericLevel (0 = Low, 1 = Medium, 2 = High, default 2). Per-frame budget is computed from profile + FPS + FOV + base drops — visible density may differ per machine. This is intentional.
5 Cone & distance culling Particles outside the view cone or beyond culling distance are deactivated or redistributed. Very large volumes can look thinner than expected. Recommendation: place weather zones along typical racing/sight corridors rather than one large global volume covering the whole map.
6 Deterministic seeds Spawn variation (position, colour, flake selection) is deterministic per zone and particle index, not random per frame. Result: stable, consistent appearance across clients.
7 QA / validation Test with cg_atmosphericLevel 0, 1, and 2 before release. Check that the volume size and placement looks correct at all three quality levels.

Scripted Objects — Full Reference

Scripted objects allow interactive, destructible, and moveable props to be placed in a map. They require a .script file and a rally_scripted_object entity in the editor.

Step 1 — Create the script file

Create a *.script file describing the object. The rally_scripted_object block defines the model, destroyed model, mass, health, bounds, and sounds.

rally_scripted_object {
    type         0
    model        "hydrant"        // direct path or named section reference
    deadmodel    "waterspray"     // shown after destruction
    moveable     0                // 1 = subject to physics
    mass         50
    health       1000             // 0 = indestructible
    mins         "-8 -8 -8"
    maxs         "8 8 8"
    presound     "sound/hydrant/metalcreak"
    hitsound     "sound/hydrant/solidmetalbang"
    destroysound "sound/hydrant/solidmetalbreak"
    postsound    "sound/hydrant/waterspray"
    gibs {
        "models/mapobjects/hydrant/piece1" -3
        "models/mapobjects/hydrant/piece2"  2
        "models/mapobjects/hydrant/piece3"  1
        "models/mapobjects/hydrant/piece4"  4
    }
}

The model and deadmodel values can either be a direct model path (e.g. "models/mapobjects/hydrant/hydrant") or the name of another section in the same script file that contains a model token — in that case the path is loaded from that section automatically.

Step 2 — Place the entity in Radiant

Select entity class rally_scripted_object and set these key/value pairs:

KeyValueNotes
scriptpath/to/filePath to the script file, without the .script extension. Required.
angle / anglesyaw / pitch yaw rollOptional orientation.
targetnamestringOptional — allows triggers to reference the object.

Step 3 — In-game behaviour

  • On map load the game reads the script path and loads the file. Objects without a valid script are discarded with a console warning.
  • moveable 1 subjects the object to the physics simulation.
  • health > 0 makes the object destructible — on death the deadmodel is shown and gibs are spawned.
  • All defined sounds (hitsound, presound, postsound, destroysound) are played automatically at the appropriate moments.

Runtime Hazard Fields

Spawned by rear-fire weapons or map scripts. Oil slicks lower friction (and can be ignited), bio sludge expands its toxic radius over time, flame trails inflict sustained burn damage to followers.

Jukebox OGG Metadata

The in-game jukebox reads optional Vorbis comments from .ogg files placed in music/jukebox/. Drop additional tracks there to extend the rotation.

TagBehaviour
TITLEShown in the jukebox overlay.
ARTISTShown alongside title as Artist — Title.
ALBUMRead and cached but not currently displayed in the overlay.

Display priority: if both ARTIST and TITLE are present the overlay shows Artist — Title. If only TITLE is set it shows the title alone. Files without tags fall back to the filename-derived display name — existing behaviour is fully preserved.

Menu music rotation uses a separate path: place any .ogg file prefixed with menumusic in baseq3r/music/ to add it to the main-menu auto-play rotation.

Racing Bot WaypointsBots navigate from .ghost replay files. The server loads the fastest ghost route, samples waypoints, and uses them for steering with a fixed lookahead and smoothed target speed. Legacy .bpd bezier paths are also supported. The full Ghost Converter and in-game recording commands (recordRacingLine start/stop/save) are planned but not yet in the production build.
Compilation

Building Q3Rally

Q3Rally uses GNU Make and is based on ioquake3. The engine directory contains all source code. Compiling produces the game binary plus QVM files for the game logic.

Quick StartClone the repo, cd engine, run make. That's it on most Linux and macOS systems.

Windows

MSYS2 Recommended
  1. Install MSYS2 with packages: git make mingw-w64-i686-gcc mingw-w64-x86_64-gcc
  2. Open MSYS2 MinGW 64-bit
  3. git clone https://github.com/Q3Rally-Team/q3rally.git
  4. cd q3rally/engine && make
WSL (Windows Subsystem for Linux)
  1. Install WSL + Ubuntu from the Windows Store
  2. sudo apt install git make gcc mingw-w64
  3. git clone https://github.com/Q3Rally-Team/q3rally.git
  4. cd q3rally/engine && make PLATFORM=mingw32

Linux

Standard Build
  1. Install: git make gcc libsdl2-dev
  2. git clone https://github.com/Q3Rally-Team/q3rally.git
  3. cd q3rally/engine && make

For a portable release bundle including SDL2: ./make-linux-portable.sh x86_64

macOS

Standard Build
  1. Install Xcode command-line tools
  2. git clone https://github.com/Q3Rally-Team/q3rally.git
  3. cd q3rally/engine && make
AppBundle
  • Modern macOS 10.9+ (Apple Silicon + x86_64): ./make-macosx-ub2.sh
  • Legacy macOS 10.5+: ./make-macosx-ub.sh
  • Single arch: ./make-macosx.sh x86_64 (or arm64)

Architecture Flags

FlagDescription
make ARCH=x86_64Compile for 64-bit x86 explicitly
make ARCH=x86Compile for 32-bit x86
make ARCH=arm64Compile for ARM64 / Apple Silicon
make PLATFORM=mingw32Cross-compile for Windows (requires mingw-w64)
make PLATFORM=darwinCross-compile for macOS (requires osxcross)

Docker

A pre-built Docker image is available — no compilation required.

# Pull
docker pull registry.gitlab.com/manalejandro/q3rally-docker/q3rally-docker:latest

# Run client
docker run --rm -it registry.gitlab.com/manalejandro/q3rally-docker/q3rally-docker:latest

# Run dedicated server
docker run --rm -it -p 27960:27960/udp \
  registry.gitlab.com/manalejandro/q3rally-docker/q3rally-docker:latest \
  +set dedicated 1 +exec server.cfg

Run Without Installing

./run.sh +set g_gametype 6 +map q3r_battlecanyon

The run.sh script builds if needed, creates symlinks to the compiled QVMs, and launches the game. Use run-debug.sh for native .so builds (faster iteration, no QVM compilation).
