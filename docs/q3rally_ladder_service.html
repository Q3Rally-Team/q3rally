Recommended Deployment

PHP Webservice

A self-contained single-file PHP service that receives match data from Q3Rally dedicated servers, stores it as JSON files, and renders a full leaderboard dashboard. No database or framework required — runs on any standard PHP 8.0+ shared hosting.

RequirementsPHP 8.0 or newer. Write access to the data/ subdirectory. No additional libraries or extensions needed.

Deployment

Shared Hosting Recommended
  1. Upload the contents of ladder_service/php_webservice/ to your webspace.
  2. Ensure PHP 8.0+ is active and the webserver has write access to the data/ folder. Grant write permissions via your hosting panel if needed.
  3. The API is now reachable at your webspace URL, e.g. https://example.com/ladder/index.php.

Many hosts set index.php as the default file. If the folder is in the document root the base URL may simply be https://example.com/.

Data Storage

Each match is stored as a single JSON file under data/<matchId>.json. The service automatically appends a receivedAt timestamp for sorting. Back up the data/ folder regularly. For very large installations a full database backend is recommended long-term.

Web Dashboard

The root page of index.php serves a full leaderboard dashboard that reads from the data/ folder and renders four ranking categories:

CategoryModesMetric
Best TimesRacing, Team Racing, SprintFastest lap/finish time per player per map
DeathmatchDM, Team DMK/D ratio, kills, deaths — sorted by ratio
ObjectiveCTF, Domination, Derby, LCSCaptures / objectives / score — mode-priority sorted
EliminationEliminationRound wins (derived from position + roster if not explicit)
Domination ScoringThe dashboard searches each scoreboard entry for objectives, objectiveScore, or objectivePoints fields. If present, that value is used as the ranking metric. Otherwise it falls back to the general score/points field.

Security & Rate Limiting

FeatureDetail
Per-server API keysManaged via register.php / admin.php. Each POST must include a valid Authorization: Bearer <key> header.
Rate limitingMax 30 requests per 60 s per IP by default (LADDER_RATE_LIMIT_MAX / LADDER_RATE_LIMIT_WINDOW).
Payload sizeMax 512 KB per POST body (LADDER_MAX_BODY_BYTES).
DeduplicationRe-posting an existing matchId returns HTTP 200 silently — no duplicate stored.
Key sanitisationThe server key is stripped from all publicly readable match payloads.

Error Responses

All errors return JSON: { "error": "..." }. File permission issues return HTTP 500. Malformed payloads return HTTP 400.

Development / Self-Hosted

Python Service

A lightweight FastAPI/SQLite service for development or self-hosted deployments. Implements the same REST interface as the PHP webservice but stores matches in a SQLite database instead of flat JSON files.

Use CaseThe Python service is best suited for local development, testing, or server environments where you control the stack. For simple shared hosting the PHP webservice is easier to deploy.

Requirements

  • Python 3.11
  • Dependencies from requirements.txt

Installation

cd ladder_service
python -m venv .venv
source .venv/bin/activate        # Windows: .venv\Scripts\activate
pip install -r requirements.txt

Start Development Server

uvicorn ladder_service.main:app --reload

By default a ladder.db SQLite file is created in the working directory. Override the path with the environment variable LADDER_DB_PATH.

Run Tests

pytest

Integration

API Reference

Both service implementations expose the same REST API. The base URL depends on your deployment.

Authentication

All POST requests require a bearer token in the Authorization header. Keys are managed server-side via register.php / admin.php (PHP) or the admin interface (Python).

Authorization: Bearer your-secret-api-key

POST /matches — Submit a Match

curl -X POST https://example.com/ladder/index.php/matches \
     -H "Content-Type: application/json" \
     -H "Authorization: Bearer your-secret-api-key" \
     -d @match.json

The payload is the JSON object built by the Q3Rally dedicated server. Required top-level fields:

FieldRequiredDescription
matchId✔Unique match identifier, e.g. srv-20240405-183011-42
mode✔Gametype string, e.g. GT_RACING
startTime / endTime✔ISO 8601 timestamps
players[*].playerId✔Persistent player identity (cl_guid hash)
players[*].rawScore✔Mode-dependent primary metric
settings.g_gametype✔Integer gametype value
mapMap name string
serverObject: name, host, build, dedicated
teamsTeam aggregate objects (team modes)
eventsOptional lap/flag/kill event log

Responses: 201 Created with { "matchId": "…" } on success. 200 OK if matchId already exists (no-op). 400 for malformed JSON. 413 if payload exceeds 512 KB. 429 if rate-limited.

GET /matches — List Matches

curl "https://example.com/ladder/index.php/matches?limit=10&mode=GT_RACING"
ParameterDefaultDescription
limit100Max results to return. Pass all for streaming output.
offset0Skip this many results (pagination).
mode—Filter by gametype string, e.g. GT_RACING.

GET /matches/{matchId} — Single Match

curl https://example.com/ladder/index.php/matches/srv-20240405-183011-42

Returns the full stored payload. The server key is stripped from the response.

DELETE /matches/{matchId}

curl -X DELETE https://example.com/ladder/index.php/matches/srv-20240405-183011-42

Returns 204 No Content on success. Requires admin auth.

GET /matches/index — Fast Leaderboard Index

curl https://example.com/ladder/index.php/matches/index

Returns only the leaderboard-relevant fields for every match (no full payloads). Used by the web dashboard to load all results efficiently. Cached for 30 s.

GET /maps/metadata & /maps/levelshots

# Map long-name lookup (from .arena files)
curl https://example.com/ladder/index.php/maps/metadata

# Available levelshot image manifest
curl https://example.com/ladder/index.php/maps/levelshots

GET/POST /players/{playerId}

curl https://example.com/ladder/index.php/players/abc123def456

Returns career profile data: rank, stats, achievements. Updated automatically on each match upload when the local client includes a profile snapshot.

Client Update System

Version Check Endpoint

The Q3Rally client checks for updates on startup by fetching a version text file from the ladder server. The check is triggered on every UI load (CL_InitUI) and runs asynchronously in the background.

Default Endpointhttps://ladder.q3rally.com/version.txt — overrideable via the cl_updateEndpoint CVar.

How the Client Works

  1. On UI init, CL_UpdateRequestLatest fires an async HTTP GET to cl_updateEndpoint.
  2. The response is parsed: first line is the version string. Leading/trailing whitespace and optional surrounding quotes are stripped. A leading v/V is optional.
  3. The remote version is compared against the local PRODUCT_VERSION. If newer: status outdated, main menu shows a download prompt. Otherwise: up_to_date.
  4. Network errors or invalid responses set status error — no prompt shown.

Serving the Version File (PHP)

The PHP webservice exposes GET /version for reading and POST /version for updating. A password protects writes.

Setting the Password

Set via environment variable (preferred) or a password file:

# Option 1: environment variable (.htaccess on shared hosting)
SetEnv LADDER_VERSION_PASSWORD "your-secret"

# Option 2: password file (auto-read if present)
echo -n "your-secret" > /path/to/ladder/version.password

Without a configured password, write requests return HTTP 403.

Read Current Version

curl https://example.com/ladder/index.php/version

Returns JSON with at least latest (or version). Pass Accept: text/plain to get a plain text response.

Update the Version

curl -X POST https://example.com/ladder/index.php/version \
     -H "Content-Type: application/json" \
     -d '{
       "password":    "your-secret",
       "version":     "v0.7c",
       "downloadUrl": "https://downloads.example.com/q3rally-v0.7c.zip",
       "message":     "Bugfix release — gametype enum fixed."
     }'

downloadUrl and message are optional and can be overwritten or cleared at any time.

Static File Alternative

If you don't need the update endpoint, a plain static text file works too — just place version.txt on any HTTPS server with the version string on the first line:

v0.7c

Point cl_updateEndpoint to its URL. The client handles both JSON and plain-text responses.
