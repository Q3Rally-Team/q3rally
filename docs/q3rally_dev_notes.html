Bot AI

Racing Bot Waypoint System

Current state, identified issues, and the planned hybrid Ghost-Converter approach for improving bot racing quality.

Current State

What Works TodayServer loads .ghost files robustly (header/frames, size limit, fallbacks). Fastest matching ghost is selected as the bot route. Bots actively use waypoints for steering direction and approximate target speed. Legacy .bpd bezier paths from map entities still load at map start.

Identified Issues

1. Track Crossing Ambiguity Open
Per tick the nearest waypoint is searched geometrically. At track overlaps / crossings this can snap to the wrong section of the circuit.

Proposal: Cache a progress index per bot (forward-only with a small search window). Optionally blend checkpoint-progress + ghost-progress.
2. Fixed-Index Lookahead Open
bestIndex + 12 is track- and sampling-dependent. Dense or sparse samples cause too-short or too-long lookahead.

Proposal: Lookahead by time offset (+400–900 ms) or distance (+20–50 m) instead of fixed count. Or adaptive scaling based on current speed.
3. No Segment Projection Open
Steering targets a future waypoint but does not project onto the current segment (i→i+1). Causes unnecessary oscillation.

Proposal: Project position onto segment, derive tangential direction. Optionally apply Catmull-Rom smoothing.
4. Target Speed from Single Segment Open
Target speed estimated only from bestIndex and bestIndex+1 — susceptible to timing jitter in ghost frames.

Proposal: Average speed over a window (5–15 segments). Use curvature lookahead to decelerate early for corners.
5. No Route Quality Metric Open
Any parseable route is used even if noisy.

Proposal: Plausibility checks on load: monotone timestamps, min/max distances, outlier filter. Optional resampling to constant time grid.

Hybrid Ghost-Converter Plan

Status: Planned — Not Yet ImplementedThe ghost converter, in-game recording commands, and flexible corridor navigation are design documents. They are not present in the current production build.

Preferred Workflow

  1. Input: ghost/<map>/<run>.ghost
  2. Tool: ghost2bpd input.ghost output.bpd
  3. Output: maps/<map>.bpd

On conversion: ghost samples are resampled and smoothed, projected onto checkpoint segments, Bézier handles estimated from tangents and curvature.

Fallback: In-Game Record & Bake

  1. recordRacingLine start
  2. Drive a reference lap
  3. recordRacingLine stop
  4. recordRacingLine save

Saves as maps/<map>.bpd in the existing parser-compatible format.

Minimal-Invasive Implementation Strategy

  1. Introduce ghostRouteIndexHint per bot
  2. Nearest search only in local window around hint (e.g. [−8, +24])
  3. Lookahead by time offset instead of fixed waypoint count
  4. Speed smoothing over multiple segments

This stays fully compatible with the existing .ghost format while gaining robustness and pace consistency.

1Progress index + search windowBiggest stability improvement immediately
2Time/distance lookaheadBetter line choice across varying track layouts
3Segment projection + smoothed target speedLess steering jitter, more natural bot movement
Investigations

Bug Investigations & Fixes

Internal post-mortems and fix documentation for non-trivial bugs.

Team Mode Entity Overflow

G_Spawn: no free entities in team game types Fixed

Symptom: Server aborts with G_Spawn: no free entities during any team-based game type. Entity dump shows hundreds of entries tagged noclass.

Root cause: CopyToBodyQue clones the extra collision bounds (frontBounds/rearBounds) from the dying vehicle so the corpse keeps its full collision hull. Body queue slots are recycled in a ring buffer — helper entities from a previous corpse may still exist when the slot is reused. The old code overwrote body->frontBounds/rearBounds pointers with new G_Spawn() results, leaving the old helpers orphaned. The leak quickly exhausted the entity pool in small-team modes with frequent deaths.

Fix: Each body-queue entry now owns two persistent helper entities created alongside the queue in InitBodyQue. When a corpse is copied into the slot the helpers are simply updated and relinked; when the corpse sinks they are unlinked instead of freed. A shared G_ResetExtraBBox helper keeps all code paths (body sinking, gibbing, etc.) consistent. Entity count stays flat regardless of death frequency.

Ghost Waypoint System Review

Waypoint ambiguity at track crossings Open
See the Bot Waypoints tab for the full analysis and proposed fixes. Root issue: nearest-waypoint search without a progress hint can snap to the wrong track section at overlaps.

v0.7c — Gametype Enum Shift

DM dedicated server cycling maps immediately Fixed in v0.7c

Symptom: A dedicated server in DM mode immediately cycles through the map rotation without waiting for players.

Root cause: GT_SPRINT was inserted into the gametype_t enum in bg_public.h without explicit integer values, shifting GT_DEATHMATCH from 6 to 7 and all team gametypes by one. A server started with g_gametype 6 (DM) was therefore running GT_ELIMINATION (a race mode), which exits immediately when no players are present.

Fix: Assigned explicit integer values to all entries in gametype_t. Non-team gametypes retain v0.6c values (0–6), GT_SPRINT = 7. Team gametypes start at 16, leaving slots 8–15 for future non-team modes. See the Server Setup tab in the manual for the migration table.

Match Reporting

Telemetry Reporting Plan

Planned payload structure and delivery mechanics for automated match reporting from dedicated servers. Describes per-mode metrics, normalization, transport, and the required new CVars.

Status: Planned — Not Yet ImplementedThe telemetry system and its CVars (sv_telemetry*) are a design document. They are not present in the current production build. The ladder service uses sv_ladderEnabled / sv_ladderUrl / sv_ladderApiKey instead.

Common Player Object

All mode payloads share this base player structure:

{
  "playerId":        "sha256:...",   // SHA-256 of cl_guid + server namespace
  "displayName":     "Player",
  "team":            "red" | "blue" | null,
  "car":             "carmodel/skin",
  "normalizedScore": 0.0,            // 0.0–1.0, scope = match
  "rawScore":        0,
  "damageDealt":     { "raw": 1234, "normalized": 0.52 },
  "damageTaken":     { "raw": 800,  "normalized": 0.33 },
  "timeAlive":       "PT12M3.542S"
}
  • Player ID: SHA-256 of cl_guid salted with sv_telemetryNamespace to avoid GUID collisions between communities.
  • Time format: absolute timestamps as RFC 3339 UTC, durations as ISO 8601 (PTmmMss.mmmS).
  • Normalization: raw metrics mapped to 0.0–1.0 per match scope. Accuracy (already a percentage) is divided by 100.

Transport & Retry

  • Async POST via libcurl (same CURLM pattern as the ladder reporter).
  • HTTP 5xx or network failure → spool to telemetry/outbox, exponential backoff (5 s start, 5 min cap).
  • HTTP 202 queued → optional polling via GET /v1/matches/{matchId} (disabled by default).
  • Spool limited to sv_telemetryMaxBatch entries under fs_homepath.
